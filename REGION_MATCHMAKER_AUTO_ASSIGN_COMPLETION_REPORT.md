### 3. ✅ 自动分配逻辑

#### 自动分配流程:
```
用户注册
    ↓
检查自动分配功能开关
    ↓
开启?
    ├─ 是 → 获取用户地区
    │        ↓
    │        查找负责该地区的红娘
    │        ↓
    │        找到?
    │        ├─ 是 → 分配给该红娘(标记为自动分配)
    │        └─ 否 → 地区优先?
    │            ├─ 是 → 随机分配红娘
    │            └─ 否 → 暂时由总管理后台
    └─ 否 → 随机分配红娘
```

#### 分配规则:
1. **自动分配功能开启**:
   - 查找负责用户所在地区的红娘
   - 如果找到,分配给该红娘
   - 如果找不到,检查地区优先开关
     - 开启: 随机分配一个红娘
     - 关闭: 暂时由总管理后台管理

2. **自动分配功能关闭**:
   - 随机分配一个红娘

3. **红娘状态**:
   - 只分配给正常状态的红娘
   - 禁用的红娘不参与分配

## 自动分配规则详解

### 规则1: 自动分配功能开启
```
用户注册
    ↓
获取用户所在地区
    ↓
查找负责该地区的红娘
    ↓
找到匹配的红娘?
    ├─ 是 → 分配给该红娘
    │      标记: 自动分配
    │      保存分配记录
    └─ 否 → 检查地区优先开关
             ├─ 开启 → 随机分配红娘
             │          标记: 自动分配
             │          保存分配记录
             └─ 关闭 → 暂时由总管理后台管理
                        标记: 总后台管理
                        保存分配记录
```

### 规则2: 自动分配功能关闭
```
用户注册
    ↓
随机分配一个红娘(正常状态)
    ↓
标记: 手动分配
    ↓
保存分配记录
```

### 规则3: 红娘筛选
- 只选择状态为"正常"的红娘
- 禁用的红娘不参与分配
- 红娘必须至少负责一个地区

### 规则4: 地区匹配
- 精确匹配用户所在地区
- 红娘负责地区包含用户地区
- 支持一红娘多地区配置

```
用户注册触发
    ↓

    ↓

    ↓

    ↓
```

### API接口设计:
```javascript
// 红娘管理

GET /api/admin/matchmakers - 获取红娘列表

// 用户分配
POST /api/admin/users/:userId/assign - 分配用户
PUT /api/admin/users/:userId/reassign - 重新分配
GET /api/admin/users/assign-records - 获取分配记录

// 功能开关
PUT /api/admin/config/auto-assign - 更新自动分配开关
PUT /api/admin/config/region-priority - 更新地区优先开关
```
